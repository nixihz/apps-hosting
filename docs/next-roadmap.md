# apps 后 MVP 功能丰富路线图

当前主干已经完成 Phase 1 ~ Phase 6 基线。后续目标从“能跑通平台”转向“把平台变好用、好管、好扩展”。

## 总目标

1. 让非平台团队也能低成本发布小应用。
2. 让管理员能清晰治理权限、环境、配置和资源。
3. 让插件生态具备可发布、可发现、可复用、可集成能力。
4. 让平台具备更接近生产环境的稳定性、安全性和可观测性。

## Phase 7：体验增强与管理台产品化

目标：把当前可用的管理页面升级成更完整的产品界面。

### 核心能力

- 应用详情页
- 应用安装向导
- 应用部署向导
- 配置编辑器
- 发布历史与回滚页面
- 日志查看器
- 指标看板

### 成功标准

- 管理员不依赖 CLI 也能完成常规应用管理。
- 开发者可以通过 Web 页面完成部署、配置和回滚。
- 普通用户可以通过首页快速访问已安装且有权限的应用。

## Phase 8：安全、认证与权限体系增强

目标：从 MVP 的请求头 / token 模式升级为可接入组织认证的权限体系。

### 核心能力

- OIDC / OAuth2 登录接入
- 用户、用户组、角色管理
- RBAC 权限模型
- 应用级授权策略
- API token 管理
- Webhook 签名校验
- secrets 加密存储
- 审计查询与导出

### 成功标准

- 平台可以接入企业统一身份系统。
- 管理员可以按用户组、租户、角色授权。
- 敏感配置不会明文暴露。

## Phase 9：部署流水线与运行时生产化

目标：让部署能力从本地目录发布升级到更贴近真实生产使用。

### 核心能力

- Git URL clone / pull
- 分支 / tag / commit 选择
- 构建队列
- 构建缓存
- 构建日志流式输出
- 发布审批
- 灰度发布
- 多副本切换
- 运行时适配器：Node / Python / Go / Java / Docker
- 进程守护与自动拉起

### 成功标准

- 应用可以直接从 Git 仓库发布。
- 构建、部署、审批、回滚都有可追踪记录。
- 后端服务具备更稳定的守护和重启策略。

## Phase 10：插件 SDK 与生态市场

目标：让插件从“能运行”升级到“可开发、可发布、可复用”。

### 核心能力

- 插件 SDK 标准化
- 插件生命周期 hook
- 插件权限声明
- 插件配置 schema
- 插件市场发布流程
- 插件版本兼容检查
- 插件评分、收藏、安装统计
- AI 工具插件模板
- 前端组件扩展点

### 成功标准

- 开发者可以按 SDK 开发插件。
- 插件能声明权限、配置、菜单、API、事件订阅。
- 平台能识别插件兼容性和风险。

## Phase 11：可观测性、告警与治理增强

目标：让平台具备可追踪、可告警、可治理的运维能力。

### 核心能力

- 持久化指标存储
- 访问趋势与错误趋势
- 应用 SLO / SLA 配置
- 健康检查策略
- 告警规则
- Webhook / 飞书 / Slack 告警
- 资源配额与用量报表
- 治理评分
- 风险检查清单

### 成功标准

- 管理员可以看到应用健康与资源使用趋势。
- 应用异常可以自动触发告警。
- 平台可以输出治理报告和整改建议。

## Phase 12：分布式、多节点与组织级部署

目标：让 apps 可以作为组织级轻应用基础设施运行。

### 核心能力

- 多节点注册
- 应用调度
- 节点健康检查
- 共享存储适配
- 外部数据库适配
- 备份恢复
- 多租户隔离增强
- 环境晋级：dev -> test -> staging -> production
- 平台自身升级策略

### 成功标准

- 平台可以跨节点管理应用。
- 应用可以按环境晋级发布。
- 平台数据可以可靠备份和恢复。

## 近期优先级

建议下一轮先做：

1. Phase 7：管理台产品化
2. Phase 8：认证与权限增强
3. Phase 9：Git URL 发布与构建队列
4. Phase 10：插件 SDK 标准化
5. Phase 11：告警与治理评分

## 新版本定位

`v0.7.0` 定位为 **Post-MVP Feature Expansion Planning Release**：

- 明确主干 MVP 已完成。
- 规划下一阶段功能丰富方向。
- 为后续持续迭代建立版本节奏。
